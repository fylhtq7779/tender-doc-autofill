# Архитектура решения

## 1. Подход: детерминированная подстановка

Система использует **детерминированную подстановку** вместо LLM: каждое поле имеет явный источник (`PROFILE`, `TENDER`, `CALC`, `SYSTEM`) и dot-path до значения. Это исключает галлюцинации, обеспечивает воспроизводимость и позволяет аудитировать каждое заполненное поле.

LLM задействован только в бонусной задаче — извлечение полей из входящего ТКП (`extract-tender`), где структура документа заранее неизвестна. Для типовых реквизитов и полей шаблонов AI не используется.

## 2. Компоненты системы

```
┌─────────────────────────────────────────────────────────────────┐
│                        python -m src generate                   │
└───────────────────────────┬─────────────────────────────────────┘
                            │
               ┌────────────▼────────────┐
               │      Generator          │  координирует процесс
               └────┬──────────┬─────────┘
                    │          │
       ┌────────────▼──┐  ┌────▼─────────────┐
       │  DataLoader   │  │  MappingLoader   │
       │  data/*.json  │  │  mappings/*.yaml  │
       └────────────┬──┘  └────┬─────────────┘
                    │          │
               ┌────▼──────────▼─────────┐
               │     DataContext          │  profile + tender +
               │  + DocumentMapping       │  calc + system + rules
               └────────────┬────────────┘
                            │
               ┌────────────▼────────────┐
               │    TemplateEngine        │  python-docx
               │  templates/*.docx        │
               └────────────┬────────────┘
                            │
               ┌────────────▼────────────┐
               │     output/*.docx        │
               └─────────────────────────┘
```

**DataLoader** — загружает три JSON-файла, валидирует через Pydantic-модели (`CompanyProfile`, `TenderData`, `CalcData`) и генерирует системные поля (`SystemData`): текущую дату, номер исходящего.

**MappingLoader** — читает YAML-файлы из `mappings/`, парсит в `DocumentMapping` (список `FieldMapping` + список `TableRowsMapping`). Файлы обрабатываются в алфавитном порядке.

**Generator** — для каждого `DocumentMapping` вызывает `TemplateEngine`, резолвит значения полей из `DataContext` и применяет трансформации.

**TemplateEngine** — работает с DOCX через `python-docx`: ищет плейсхолдеры в параграфах и ячейках таблиц, заполняет строки таблиц по индексу.

## 3. Маппинг-конфигурация

YAML выбран как формат конфигурации по трём причинам: читаем без специальных инструментов, поддерживает комментарии, не требует экранирования кириллицы (в отличие от JSON).

Каждый YAML-файл описывает один документ полностью:

```yaml
document:
  name: "Анкета участника"
  template: "templates/01_anketa.docx"
  output_name: "01_Анкета_участника.docx"

fields:
  - placeholder: "[ИНН]"
    source: PROFILE        # откуда брать данные
    path: "company.inn"    # dot-path (поддерживает вложенность и items[0])
    required: true         # true → [НЕ УКАЗАНО] при отсутствии; false → ""
    transform: null        # date_long | money | null

table_rows:
  - table_idx: 1           # индекс таблицы в документе (0-based)
    source: CALC
    items_path: "items"    # путь к массиву в источнике
    row_start: 1           # первая строка данных (пропуская заголовок)
    columns:
      0: { path: "line_no" }
      3: { path: "unit_price_with_delivery_w_vat", transform: "money" }
```

Логика резолвинга значений: `source` → `model.model_dump()` → `resolve_dot_path(data, path)` → `apply_transform(value, transform)`. Колонки таблицы могут переопределять `source` и `items_path` для работы с разными источниками в одной таблице (например, TENDER + CALC).

## 4. Работа с DOCX

DOCX — это ZIP-архив с XML. Ключевая особенность python-docx: текст одного абзаца может быть разбит на несколько объектов `Run` (Word делает это при редактировании, изменении форматирования). Плейсхолдер `[ИНН]` может оказаться разбит на три Run-а: `[`, `ИНН`, `]`.

**Решение в проекте:** шаблоны подготовлены так, что плейсхолдеры целиком находятся в одном Run. Замена выполняется через `run.text = run.text.replace(placeholder, value)` с сохранением всех атрибутов форматирования Run-а (шрифт, размер, жирный, курсив).

Для ячеек таблиц используется `_set_cell_text`: находит первый Run с явно заданным шрифтом (не наследованным), устанавливает ему новый текст, удаляет лишние Run-ы. Это сохраняет форматирование из шаблона.

## 5. Обработка ошибок

| Ситуация | Поведение |
|----------|-----------|
| `required: true`, значение не найдено | Подставляется `[НЕ УКАЗАНО]`, в лог пишется `WARNING` с указанием поля и источника |
| `required: false`, значение не найдено | Подставляется пустая строка |
| Невалидный JSON | `pydantic.ValidationError` с описанием поля |
| JSON-файл не найден | `FileNotFoundError` с именем файла |
| Строка таблицы вне диапазона | `WARNING` в лог, строка пропускается |
| Невалидный реквизит (ИНН, БИК) | `WARNING` в лог, значение сохраняется как есть |
| Неизвестная трансформация | `WARNING` в лог, возвращается `str(value)` |

Все ошибки фатального уровня пробрасываются через `click.ClickException` и выводятся пользователю с понятным сообщением.

## 6. Расширяемость

**Добавить новый документ** = создать шаблон + YAML-файл. Код не меняется:

```
templates/05_dogovor.docx      ← шаблон с плейсхолдерами
mappings/05_dogovor.yaml       ← маппинг полей
```

`MappingLoader.load_all()` автоматически подхватывает все `*.yaml` файлы из директории в алфавитном порядке.

**Масштабирование на 10+ документов:**
- Каждый документ изолирован в своём YAML, нет общего состояния
- Генерация линейная — `O(n)` по числу документов, добавление нового не влияет на другие
- При необходимости параллельной генерации `generate_all()` можно заменить на `ThreadPoolExecutor` без изменения остального кода

**Где может помочь AI:**
- Команда `extract-tender` уже реализована — regex-парсинг произвольного DOCX ТКП с извлечением структурированных данных в формат `tender.json`
- Потенциально: автоматическое определение плейсхолдеров в новом шаблоне и генерация черновика YAML-маппинга по образцу
