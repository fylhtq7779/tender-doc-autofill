# Архитектура решения

## 1. Подход: правила вместо AI

Система заполняет документы по явным правилам, а не с помощью нейросетей.

Для каждого поля в документе заранее прописано:
- **откуда** брать данные (из какого JSON-файла)
- **какое именно** поле взять (путь вроде `company.inn`)
- **как** отформатировать (как дату, как сумму, или как есть)

AI не используется, потому что для подстановки данных из структурированных JSON-файлов в шаблоны он не нужен:

- **Предсказуемость.** Одни и те же данные всегда дают одинаковый результат.
- **Скорость.** Генерация всех документов занимает доли секунды. Нет зависимости от внешних сервисов.
- **Автономность.** Работает локально, без интернета, без API-ключей, без подписок.
- **Проверяемость.** Для любого поля можно точно сказать: откуда оно, из какого JSON, по какому пути.

Команда `extract-tender` (извлечение данных из входящего документа заказчика) тоже работает без AI. Она находит в документе строки с известными метками ("Заказчик:", "Номер закупки:" и т.д.) и достаёт из них значения. Если в будущем потребуется разбирать документы с непредсказуемой структурой, AI можно подключить.

## 2. Из чего состоит система

```
    JSON-файлы              YAML-правила              Word-шаблоны
    (данные)              (что куда ставить)         (с метками [ИНН])
        │                       │                          │
        ▼                       ▼                          │
   DataLoader              MappingLoader                   │
   (читает JSON,           (читает YAML,                   │
    проверяет               разбирает правила)             │
    структуру)                  │                          │
        │                       │                          │
        └───────────┬───────────┘                          │
                    ▼                                      │
               Generator                                   │
            (для каждого поля:                             │
             берёт значение из JSON,          TemplateEngine
             форматирует,            ───────► (открывает Word,
             передаёт в движок)                находит метку,
                                               заменяет на значение,
                                               сохраняет форматирование)
                                                     │
                                                     ▼
                                              output/*.docx
                                           (готовые документы)
```

**DataLoader** - читает три JSON-файла (профиль компании, данные тендера, расчёт цены), проверяет что все нужные поля на месте. Плюс добавляет системные поля: текущую дату, номер исходящего.

**MappingLoader** - читает YAML-файлы с правилами из папки `mappings/`. Каждый файл описывает один документ: какие метки искать, откуда брать данные, как форматировать.

**Generator** - координатор. Для каждого документа: берёт данные из JSON, смотрит правила из YAML, находит нужное значение, форматирует (например число в "432 600,00"), и передаёт в TemplateEngine.

**TemplateEngine** - работает с Word-файлами: открывает шаблон, находит метки вроде `[ИНН]` или `[Предмет закупки]`, заменяет на реальные значения. При этом сохраняет оформление из шаблона (шрифт, размер, жирный текст).

## 3. Правила заполнения (YAML)

Правила хранятся в текстовых файлах формата YAML (читаются без специальных программ, можно открыть в блокноте).

Пример - как описано заполнение ИНН в анкете:

```yaml
fields:
  - placeholder: "[ИНН]"       # эту метку ищем в Word-шаблоне
    source: PROFILE             # данные берём из company_profile.json
    path: "company.inn"         # внутри JSON идём: company → inn
    required: true              # если данных нет - напишем [НЕ УКАЗАНО]
```

Для таблиц (например, список товаров в предложении о цене) используется секция `table_rows` - она описывает, из какого массива в JSON брать строки и какие колонки чем заполнять.

YAML выбран вместо JSON потому что:
- Можно писать комментарии (объяснять зачем какое поле)
- Русский текст не превращается в `\u0418\u041d\u041d`
- Проще читать человеку

## 4. Как система работает с Word-файлами

Word-документ (.docx) внутри - это ZIP-архив с XML-файлами. Библиотека python-docx (инструмент для работы с Word из Python) позволяет открывать документ, находить текст и заменять его.

Важный нюанс: Word может разбить одно слово на несколько внутренних блоков (Run). Например, если в шаблоне часть текста жирная, а часть нет. Наши шаблоны подготовлены так, что каждая метка (`[ИНН]`, `[Предмет закупки]`) целиком находится в одном блоке - это упрощает замену.

При замене метки на значение сохраняется всё оформление: шрифт, размер, жирность, курсив. Если в шаблоне `[ИНН]` написано шрифтом Times New Roman 10pt, то и `7705123456` будет тем же шрифтом.

## 5. Что происходит когда данных нет

| Ситуация | Что делает система |
|----------|-------------------|
| Поле обязательное, данных нет | Пишет `[НЕ УКАЗАНО]` + предупреждение в консоль |
| Поле необязательное, данных нет | Оставляет пустым |
| ИНН/БИК невалидный (неправильное количество цифр) | Предупреждение в консоль, но значение всё равно подставляется |
| JSON-файл не найден | Понятное сообщение об ошибке |
| JSON-файл битый (невалидный формат) | Понятное сообщение с указанием какое поле сломано |
| Строк в таблице больше чем в шаблоне | Лишние строки пропускаются с предупреждением |

Главный принцип: система никогда не выдумывает данные. Лучше честно написать `[НЕ УКАЗАНО]`, чем подставить что-то от себя.

## 6. Расширяемость

**Добавить новый документ** - два файла, ноль изменений в коде:

1. Word-шаблон с метками → `templates/05_dogovor.docx`
2. YAML с правилами → `mappings/05_dogovor.yaml`

Система автоматически находит все YAML-файлы в папке `mappings/` и генерирует документ для каждого.

Это доказано на практике: 4-й документ (гарантийное письмо) добавлен именно так - шаблон + правила, без единой строки нового кода.

**Масштабирование на 10+ документов:**
- Каждый документ описан отдельным YAML-файлом, они не зависят друг от друга
- Добавление нового документа не влияет на существующие
- Генерация идёт последовательно, можно при необходимости распараллелить

**Где в будущем может помочь AI (сейчас не используется нигде):**
- Разбор входящих документов с непредсказуемой структурой. Сейчас `extract-tender` ищет в документе конкретные текстовые метки ("Заказчик:", "Номер закупки:" и т.д.), что работает для стандартных документов. Если структура документа непредсказуема, можно подключить AI для понимания контекста.
- Автоматическое создание черновика YAML-правил для нового шаблона по его содержимому.

## 7. Извлечение данных из входящего документа

Когда заказчик присылает запрос на поставку (ТКП) в формате Word, менеджеру приходится вручную переносить из него данные: номер закупки, позиции товаров, сроки.

Команда `extract-tender` автоматизирует это. Она открывает Word-документ и ищет в нём строки с известными метками: "Заказчик:", "Номер закупки:", "Место поставки:" и т.д. Из таблицы товаров достаёт позиции с ценами и количеством. Всё найденное сохраняет в JSON-файл.

Этот JSON потом используется как входные данные для генерации документов. То есть полная цепочка: входящий документ от заказчика -> JSON с данными -> заполненные документы для отправки заказчику.
